# Role Definition
You are a top-tier AI algorithm engineer specializing in PyTorch, the HuggingFace `diffusers` library, `peft` (LoRA), and the application of Multi-Modal Large Language Models (MLLMs like Qwen2-VL). Please help me build a code framework from scratch for "CAD Reverse Modeling Sequence Generation" based on a Multi-View Diffusion Model.

# Project Overview
The goal of this project is to implement an **Autoregressive Multi-View Structured Step Generator**.
- **Input:** A single final appearance image of a CAD part from a specific view (`I_final`).
- **Output:** An ordered sequence of "full-view modeling steps" $[G_1, G_2, \dots, G_n]$.
- Each step $G_k$ is a concatenated **$4 \times 8$ image grid** (32 sub-images in total):
  - **4 Rows (Components):** Prev Depth Map, Sketch Plane Mask, Reference Mask, Result Wireframe.
  - **8 Columns (View Angles):** V3d_XposYposZpos, V3d_XposYposZneg, V3d_XposYnegZpos, V3d_XposYnegZneg, V3d_XnegYposZpos, V3d_XnegYposZneg, V3d_XnegYnegZpos, V3d_XnegYnegZneg.

# Core Tech Stack
- Deep Learning Framework: `torch`, `torchvision`
- Diffusion Models: `diffusers` (Using SDXL or Flux.1 as the Base Model)
- Parameter-Efficient Fine-Tuning: `peft` (LoRA)
- Tensor Operations: `einops` (Crucial for handling multi-view grid reshapes and attention)

# Correction: Misunderstanding of `result_frame.png`

There is a critical misunderstanding in how the system interprets `result_frame.png` in the project. 

Currently, the script/System Prompt assumes `result_frame.png` is the wireframe of the *entire* CAD part after the step is completed. **This is incorrect.**

**Correction:** `result_frame.png` represents ONLY the **local feature** or **delta geometry** created/modified in this *specific* modeling step. It does NOT show the rest of the existing part.

**Example to understand:**
If the existing model is a rectangular box, and the current step is extruding a cylinder on its top surface:
- `result_frame.png` will ONLY show the wireframe of the newly created cylinder. It will NOT show the rectangular box.
- The Red lines will be the circular sketch on the top surface.
- The Blue line will be the termination face of the cylinder.
- The Green lines will be the generating lines (sides) of the cylinder.

# System Architecture & Workflow Handover: Single-View CAD Reverse Modeling MVP (Minimum Viable Product)

## 1. Project Overview
We are building a "Vision-Based CAD Modeling Step Reverse Generation System" (Single-View MVP). The core philosophy is **Semantic-Geometric Decoupling**: 
- A Multi-Modal LLM (Qwen2.5-VL) acts as the **Planner**, deducing topological operations (text prompts).
- A Diffusion Model (SDXL) acts as the **Renderer**, generating pixel-perfect geometry (masks and wireframes).

---

## 2. Experimental Workflow (The 4 Phases)

Please review, check, and update the codebase to ensure it strictly follows these 4 phases. Remove any legacy code related to multi-view (e.g., 4x8 grids, cross-view attention) and ensure standard `diffusers` and `transformers` APIs are used.

### Phase 1: Data Auto-Labeling (Qwen2.5-VL Inference for Dataset Creation)
**Goal:** Generate ground-truth text prompts (`prompt.txt`) for SDXL training.
- **Inputs:** 1. `final_snapshot.png` (Global Context)
  2. `prev_depth_map.png` (Clean generation base)
  3. `operation_param.json` (Factual Constraints)
  4. `overlayed_all.png` (Target of the Painter)
- **Prompting Rule:** You can provide QWen with a prompt similar to this: "A painter has been given two images ‚Äî prev_depth_map.png, final_snapshot.png. You need to provide them with a description that enables them to paint overlayed_all.png. Your description MUST NOT include numerical dimensions or internal CAD element IDs".
- **Output:** `prompt.txt` saved in the step directory.

### Phase 2: Diffusion Model Fine-Tuning (SDXL + ControlNet + IP-Adapter)
**Goal:** Train SDXL to render the geometric action based on the semantic prompt.
- **Architecture:** Standard `StableDiffusionXLControlNetPipeline`.
- **Inputs (Conditioning):**
  1. **ControlNet:** `prev_depth_map.png` (Converted to 3-channel RGB. Provides the clean geometric base).
  2. **IP-Adapter:** `final_snapshot.png` (Provides global shape reference).
  3. **Text Encoder:** `prompt.txt` (Generated from Phase 1).
- **Target (Loss Calculation):** `overlayed_all.png` of the CURRENT step.
- **Task:** The model learns to draw the yellow/cyan masks and red/green/blue wireframes precisely onto the clean depth map.

### Phase 3: MLLM Fine-Tuning (Qwen2.5-VL LoRA/SFT)
**Goal:** Train Qwen to act as the autoregressive planner during actual inference, where `operation_param.json` is NO LONGER available.
- **Inputs:**
  1. `final_snapshot.png` (Target goal).
  2. `prev_state_image` (This could be the `prev_depth_map.png` or the `overlayed_all.png` from the *previous* step, representing the current status).
- **Target:** The `prompt.txt` for the NEXT immediate operation.
- **Task:** Qwen learns to compare the current state with the final state and deduce the next logical CAD operation without needing the ground-truth JSON.

### Phase 4: Autoregressive Inference Loop (The Final Pipeline)
**Goal:** End-to-End reverse engineering of a novel CAD part.
- **Initialization:** Given $I_{final}$ and an base $depth_0$ (main_ref_depth.png).  
- **Step 1 (Plan):** Qwen (from Phase 3) takes $I_{final}$ and $depth_k$, predicting $Prompt_{k+1}$.
- **Step 2 (Render):** SDXL (from Phase 2) takes $I_{final}$, $depth_k$, and $Prompt_{k+1}$, generating $G_{k+1}$ (`overlayed_all.png`).
- **Step 3 (Execute):** A downstream CAD parser (outside the scope of these neural networks) reads $G_{k+1}$, performs the boolean operation, and renders a new clean $depth_{k+1}$.
- **Loop:** Repeat until Qwen outputs a `<STOP>` token or the depth matches the final snapshot.

# System Architecture & Workflow Handover: Single-View CAD Reverse Modeling MVP (Minimum Viable Product)

## 1. Project Overview
We are building a "Vision-Based CAD Modeling Step Reverse Generation System" (Single-View MVP). The core philosophy is **Semantic-Geometric Decoupling**: 
- A Multi-Modal LLM (Qwen2.5-VL) acts as the **Planner**, deducing topological operations (text prompts).
- A Diffusion Model (SDXL) acts as the **Renderer**, generating pixel-perfect geometry (masks and wireframes).

---

## 2. Experimental Workflow (The 4 Phases)

Please review, check, and update the codebase to ensure it strictly follows these 4 phases. Remove any legacy code related to multi-view (e.g., 4x8 grids, cross-view attention) and ensure standard `diffusers` and `transformers` APIs are used.

### Phase 1: Data Auto-Labeling (Qwen2.5-VL Inference for Dataset Creation)
**Goal:** Generate ground-truth text prompts (`prompt.txt`) for SDXL training.
- **Inputs:** 1. `final_snapshot.png` (Global Context)
  2. `prev_depth_map.png` (Clean generation base)
  3. `operation_param.json` (Factual Constraints)
  4. `overlayed_all.png` (Target of the Painter)
- **Prompting Rule:** You can provide QWen with a prompt similar to this: "A painter has been given two images ‚Äî prev_depth_map.png, final_snapshot.png. You need to provide them with a description that enables them to paint overlayed_all.png. Your description MUST NOT include numerical dimensions or internal CAD element IDs".
- **Output:** `prompt.txt` saved in the step directory.

### Phase 2: Diffusion Model Fine-Tuning (SDXL + ControlNet + IP-Adapter)
**Goal:** Train SDXL to render the geometric action based on the semantic prompt.
- **Architecture:** Standard `StableDiffusionXLControlNetPipeline`.
- **Inputs (Conditioning):**
  1. **ControlNet:** `prev_depth_map.png` (Converted to 3-channel RGB. Provides the clean geometric base).
  2. **IP-Adapter:** `final_snapshot.png` (Provides global shape reference).
  3. **Text Encoder:** `prompt.txt` (Generated from Phase 1).
- **Target (Loss Calculation):** `overlayed_all.png` of the CURRENT step.
- **Task:** The model learns to draw the yellow/cyan masks and red/green/blue wireframes precisely onto the clean depth map.

### Phase 3: MLLM Fine-Tuning (Qwen2.5-VL LoRA/SFT)
**Goal:** Train Qwen to act as the autoregressive planner during actual inference, where `operation_param.json` is NO LONGER available.
- **Inputs:**
  1. `final_snapshot.png` (Target goal).
  2. `prev_state_image` (This could be the `prev_depth_map.png` or the `overlayed_all.png` from the *previous* step, representing the current status).
- **Target:** The `prompt.txt` for the NEXT immediate operation.
- **Task:** Qwen learns to compare the current state with the final state and deduce the next logical CAD operation without needing the ground-truth JSON.

### Phase 4: Autoregressive Inference Loop (The Final Pipeline)
**Goal:** End-to-End reverse engineering of a novel CAD part.
- **Initialization:** Given $I_{final}$ and an base $depth_0$ (main_ref_depth.png).  
- **Step 1 (Plan):** Qwen (from Phase 3) takes $I_{final}$ and $depth_k$, predicting $Prompt_{k+1}$.
- **Step 2 (Render):** SDXL (from Phase 2) takes $I_{final}$, $depth_k$, and $Prompt_{k+1}$, generating $G_{k+1}$ (`overlayed_all.png`).
- **Step 3 (Execute):** A downstream CAD parser (outside the scope of these neural networks) reads $G_{k+1}$, performs the boolean operation, and renders a new clean $depth_{k+1}$.
- **Loop:** Repeat until Qwen outputs a `<STOP>` token or the depth matches the final snapshot.

3. File Definitions
  main_ref_depth.png: A depth map containing only the primary reference plane, representing the state prior to modeling.

3.1 Sequence-Level Files
Located in the root directory of the [CAD_PART_ID]_[VIEW_SUFFIX] folder, representing the final global state of the part.

  final_snapshot.png: The color rendered image of the final part from this specific view. Purpose: Serves as the Global Context, fed into the MLLM to help it understand the macroscopic location of local features; also used as the reference input for the Diffusion model's IP-Adapter.

3.2 Step-Level Files
Located within the roll_back_index_{i} folders, representing the state changes at the i-th operational step.

  prev_depth_map.png: The depth map (grayscale) of the existing model before the current modeling operation is executed. Purpose: Serves as the spatial condition input for the ControlNet in the Diffusion model, providing a "clean geometric base".

  operation_param.json: The ground-truth parameters of the current modeling step (including operation type modeling_type, depth, construct_type, etc.). Purpose: Provides absolutely accurate underlying logic for the MLLM to generate training prompts, eliminating visual ambiguity.

  overlayed_all.png: (Core File) The 4-in-1 Alpha blended composite image. A human-vision-friendly image created by overlaying the base depth map, sketch plane mask, reference mask, and colored incremental wireframe. Purpose: Serves as the "local context" image for the MLLM to understand the current operational step, and also functions as the Target image for the Diffusion model's image generation.

4. Global Core Rules & Visual Anchors
Rule A: Color Mask Encoding
  In overlayed_all.png, colors represent unique topological, geometric, and physical meanings:

  Semi-Transparent Yellow masked area: Sketch Plane.

  Semi-Transparent Cyan masked area: Reference Geometry (e.g., Path / Axis of Revolution).

  Red (Solid Line): Reference 2D Sketch used in the current operation.

  Green (Solid Line): Edges of the newly added solid entity in this step (Added Solid Entity / construct_type: "NEW", "ADD", ...).

  Magenta (Solid Line): Edges of the entity cut or removed in this step (Removed Entity / construct_type: "CUT", "REMOVE", ...).

  Blue (Solid Line): The termination face contour of the extrusion/sweep in this step (Termination Face).
